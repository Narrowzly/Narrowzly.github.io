解决老大难问题HashMap

最近对底层的东西产生了浓重的兴趣。前几天看了关于HashMap的文章，发现自己真的一无所知简直是个垃圾。

首先大家都知道的HashMap相对于HashTable的key和value都可以为null值，但是它是不安全的，因为它本身内部并没有像HashTable那样采用synchronized方法。当然如果你想同步的话可以这么做
Map m = Collections.synchronizedMap(new HashMap(...))

当然既然它叫HashMap顾名思义它以hashing原理工作，在使用
put(K key,V value)方法的时候会调用
putVal(int hash, K key, V value, boolean onlyIfAbsent,boolean evict)（存在table里面本质就是个散列表）
1.其中hash是hash(key)的值这个函数具体是这样的

至于后面为什么要先获得key的hashcode异或h无符号右移16位为什么要这么做呢，其实这是一个扰动函数，它是用来混合高低位特性，加大低位随机性，减少碰撞次数。
具体解释参考胖胖大神的解释
2.onlyIfAbsent是否会被更新，evict如果是fasle 就说明Node[] table正在被建立（初始大小16位，通过hash&table.length-1取得对应的index值）

既然知道了put具体是怎么工作的，我们来讲一讲get(Object key)，本质上是
final Node<K,V> getNode(int hash, Object key),hash如同上面所得到，但是如果key不同但是我们的hash一样怎么办，我们可以遍历那个链表(其实链表大小大于8会自动变成红黑树)找到我们要的值，那怎么个找法呢其实用到了key.equals(k)来判断。
整理过程过程如下
1.调用get(Object key)
2.调用 getNode(int hash, Object key)通过hash(key)值找到具体哈希桶的位置
3.通过key.equals(k)遍历找到具体的Value如果变成了红黑树的话getTreeNode(hash, key)找到具体的值前提是有下一个

最后来讲一讲默认的负载因子loadFactor0.75很简单就是当一个map填满了75%的bucket时候，和其它集合类(如ArrayList等)一样，将会创建原来HashMap大小的两倍的bucket数组，来重新调整map的大小，并将原来的对象放入新的bucket数组中。这个过程叫作rehashing，因为它调用hash方法找到新的bucket位置。
